# 拖拉机AI出牌策略总结

> 基于 `advanced-ai.js` 代码分析，核心原则：**让对手得最少的分**

---

## 一、牌力评估系统

### 1.1 单张牌价值 (`evaluateCardValue`)

| 牌类型 | 价值 | 说明 |
|--------|------|------|
| 大王 | 100 | 最高价值 |
| 小王 | 99 | |
| 主级牌 (主花色+当前级) | 98 | 如打2时，红桃2 |
| 副级牌 (非主花色+当前级) | 97 | 如打2时，黑桃2 |
| 主牌 | 50 + RANK_ORDER | RANK_ORDER: 3=0, 4=1 ... K=10, A=11, 2=12 |
| 副牌 | RANK_ORDER + (有分+15) | 分牌(5/10/K)额外+15 |

| 因素 | 权重 | 说明 |
|------|------|------|

| 大王 | ×20 | |
| 小王 | ×15 | |
| 主级牌 | ×12 | |
| 对子 | ×10 | |

## 二、局势分析系统 (`analyzeGameState`)

每轮出牌前分析以下信息：

| 指标 | 说明 |
|------|------|
| `leadIsTeammate` | 首家是否是队友 |
| `isLastPlayer` | AI是否最后一个出牌 |
| `currentChopper` | 场上是否有人将吃（及其seat） |
| `currentChopperTrumps` | 将吃者出的主牌（用于超将吃比较） |
| `winnerIsTeammate` | 当前赢家是否是队友 |
| `currentWinnerCards` | 当前赢家的牌（用于`isPlayBeating`判断） |
| `trickPoints` | 当前trick中的分牌总价值 |
| `opponentTrickPoints` | 对手在当前trick中贴的分牌价值 |

## 三、首家出牌策略 (`selectLeadPlay`)

### 3.1 对手缺门感知

通过 `CardTracker.isPlayerVoidInSuit(seat, suit)` 检测对手缺门情况：
- 两个对手都缺门：该花色评分 -30（非常危险，容易被将吃）
- 一个对手缺门：该花色评分 -15（较危险）
- 无对手缺门：评分不变（安全）

### 3.2 手牌数量因子

- 手牌 ≤ 4张：评分 +5（后期更激进，用大牌抢分）
- 手牌 ≥ 20张：评分 -5（前期更保守，保存实力）
- 其他：评分不变

### 3.3 出牌优先级

**核心原则：不能确保最大的情况下，不出分牌(K=10分, 10=10分, 5=5分)**

| 优先级 | 策略 | 评分 | 说明 |
|--------|------|------|------|
| 1 | 副牌A对子 | 120 | 确定最大的牌，先出 |
| 2 | 副牌A单张 | 110 | 确定最大的牌 |
| 3 | 副牌K对子(确保最大) | 105 | 有A保护，或A已出过 |
| 4 | 副牌K单张(确保最大) | 100 | 有A保护，或A已出过 |
| 5 | 副牌拖拉机 | 95 | 长套连续对子 |
| 6 | 长套副牌甩牌 | 80 | 4+张且最大是A/K |
| 7 | 其他副牌对子 | 75 | 非分牌对子(Q,J等) |
| 8 | 副牌非分牌(Q,J,9,8,7,6,4,3) | 60 | 从大到小出，安全无分牌损失 |
| 9 | 副牌K对子(无A保) | 50 | 不安全，等有A再出 |
| 10 | 副牌K单张(无A保) | 45 | 不安全，等有A再出 |
| 11 | 副牌10/5(有A保) | 55 | 有A保护时可出 |
| 12 | 副牌10/5(无A保) | 35 | 不安全，等有A再出 |
| 13 | 主牌对子/拖拉机 | 55-60 | 排除大王 |
| 14 | 小主牌清主 | 50 | 排除大王 |
| 15 | 大王 | 20 | 最低优先级，保留保底 |

**分牌保护机制**：
- K：需要同花色有A（手中或已出过）才安全出，否则评分45
- 10：需要同花色有A才安全出，否则评分35
- 5：需要同花色有A才安全出，否则评分35
- A已出过后，K/10/5变为该花色最大牌，可以安全出

**非分牌出牌顺序**：从大到小出(Q>J>9>8>7>6>4>3)，更可能赢得出牌权。

## 四、跟牌策略 (`selectFollowPlay`)

### 4.1 四种情况分支

```
队友出牌 + 队友在赢 → selectTeammateWinningPlay
队友出牌 + 对手在赢 → selectTeammateLosingPlay
对手出牌 + 队友在赢 → selectOpponentLeadingButTeammateWinningPlay
对手出牌 + 对手在赢 → selectOpponentWinningPlay
```

### 4.2 情况1：队友出牌 + 队友在赢 (`selectTeammateWinningPlay`)

**末家（队友确定最大）**：
- 贴同花色分牌 (95)
- 没同花色时：垫副分(90) > 垫最小副牌(70)
- 有同花色时：跟最小同花色(65)
- **不浪费主牌将吃**（队友已赢，主分将吃无意义）

**非末家**：
- 跟主牌：最小非分主牌(82) > 最小分主牌(65)
- 跟副牌：最小非分牌(82) > 最小分牌(65)
- 没同花色：将吃(85+对手分×3) > 垫分牌(80) > 垫最小牌(75)

**原则**：队友在赢时出最小牌，不浪费大牌；非末家避免出分牌（防对手管住得分）。

### 4.3 情况2：队友出牌 + 对手在赢 (`selectTeammateLosingPlay`)

- 有同花色：找能管住对手的牌（评分70+牌值），出最小能赢的
- 跟主牌非末家：有分牌时出最大保护，无分牌时出最小节省
- 跟不了：出最小牌(40)
- 没同花色：将吃(80+分×3) > 垫最小牌(50)

### 4.4 情况3：对手出牌 + 队友在赢 (`selectOpponentLeadingButTeammateWinningPlay`)

**末家（队友确定最大）**：
- 同情况1的末家策略（不浪费主牌将吃）

**非末家**：
- 跟主牌：最小非分主牌(82) > 最小分主牌(65)
- 跟副牌：最小非分牌(75) > 最小分牌(60)
- 没同花色：将吃(80+对手分×3) > 垫副分(78) > 垫最小副牌(70)

**原则**：不要破坏队友优势，出最小牌；有分牌时将吃接管。

### 4.5 情况4：对手出牌 + 对手在赢 (`selectOpponentWinningPlay`)

- 有同花色：找能管住对手的牌（评分90+牌值），跟主牌非末家有分牌时出最大保护，无分牌时出最小节省
- 跟不了：出最小牌(30+牌值)
- 没同花色：将吃(80+分×3) > 垫最小牌(40)

## 五、将吃(Chop)策略

### 5.1 将吃候选生成 (`generateChopCandidates`)

**非末家**：
- trick有分牌：出最大主牌将吃（保护分数）
- trick无分牌：出最小主牌将吃（节省实力）

**末家 + 场上有人将吃**：
- 队友将吃了 → 不再管，垫牌
- 对手将吃了 → 尝试超将吃（出更大的主牌）
- 超不了 → 垫最小非分副牌(60) > 最小非分主牌(55) > 最小主牌(50)

**末家 + 无人将吃**：
- 优先用主分牌将吃（+5分奖励）
- 没主分 → 最小主牌

### 5.2 牌型匹配

将吃时严格匹配首家牌型：
- 单张 → 单主将吃
- 对子 → 对主将吃
- 拖拉机 → 拖拉机主将吃
- 找不到匹配牌型时退回垫牌

### 5.3 将吃评分

```
将吃评分 = baseScore + trickPoints × 3
```

场上分牌越多，将吃越优先。例如对手贴30分时，将吃评分 = 80 + 90 = 170。

## 六、扣底策略 (`selectBottomCards`)

| 优先级 | 操作 | 说明 |
|--------|------|------|
| 高 | 扣非主分牌(10/K/5) | 分牌危险，避免被对手抠底 |
| 中高 | 扣主牌分牌(主5/主10/主K) | 被抠底时翻倍扣分，比非分主牌更危险 |
| 中 | 扣非主小牌 | 保留大牌 |
| 低 | 扣主牌非分牌 | 主牌尽量保留 |

**保护机制**：不拆对子（对子优先级-20）

## 七、亮主策略 (`decideBid`)

| 条件 | 行动 |
|------|------|
| 2+同花色级牌 + 强度≥30 | 亮主 |
| 1级牌 + 1王 + 强度≥30 | 用王反主 |
| 2+王 + 强度≥25 | 无主 |

## 八、保底机制

- 所有策略函数最终通过 `validatePlay` 验证
- 验证失败时进入 `generateFallbackCandidates` 备选
- 最终保底：出 `hand[0]`

## 八.5、底牌保护策略（保底意识）

### 8.5.1 核心概念

- 底牌分数（`bottomCardPoints`）：庄家扣底时放入的分牌总值
- 保护等级（`bottomProtection`）：0=无需保护，1=中等保护，2=强保护
- 最后一轮赢家获得底牌分数；闲家抠底时分数翻倍（单张×2，对子×4，拖拉机×6）

### 8.5.2 保护等级判定

**庄家方**（可见底牌）：
| 条件 | 保护等级 |
|------|----------|
| 后期(≤4轮) + 底分≥30 | 2（强保护） |
| 后期(≤4轮) + 底分≥15 | 1（中保护） |
| 底分≥40（前期大底） | 1（中保护） |

**闲家方**（根据已出牌估算底分）：
| 条件 | 保护等级 |
|------|----------|
| 后期(≤4轮) + 估算底分≥25 | 2（强保护） |
| 后期(≤4轮) + 估算底分≥10 | 1（中保护） |

### 8.5.3 对出牌策略的影响

**庄家方（保护底牌）**：
- 大王评分：20 → 75（强保护）/ 45（中保护）→ 积极出牌抢赢
- 小王评分：50 → 70（强保护）/ 60（中保护）→ 积极出牌抢赢

**闲家方（抢底）**：
- 大王评分：保持20 → 保留用于最后一轮抢底
- 小王评分：50 → 35（强保护）→ 保留用于最后一轮抢底
- 小主牌清主评分：50 → 35（强保护）→ 保留大牌

**跟牌策略**：
- 闲家方有强保底需求时，跟主牌始终出最小能赢的牌（保留大牌用于最后一轮）

## 九、核心设计原则

1. **最小化对手得分**：所有决策以此为最终目标
2. **非末家不出分牌**：防对手管住得分
3. **末家安全送分**：队友确定最大时贴分牌，不浪费主牌将吃
4. **K保护**：无A保护时降级K的价值
5. **大王保底**：大王评分极低(20)，尽量不出（底分多时例外）
6. **将吃优先级随分牌提升**：trickPoints越多，将吃越重要
7. **将吃根据分牌决定大小**：有分牌出最大保护，无分牌出最小节省
8. **对手缺门感知**：出牌前检查对手缺门情况，避免被将吃
9. **手牌数量因子**：后期更激进，前期更保守
10. **底牌保护意识**：庄家方底分多时用大牌保底，闲家方估算底分多时保留大牌抢底

## 十、已修复问题记录

### ✅ 问题1：非末家跟主牌时"出最大能赢的牌"过于激进
- **修复**：trick有分牌时出最大保护，无分牌时出最小节省
- **影响**：减少大牌浪费

### ✅ 问题2：非末家将吃时"出最大主牌"过于激进
- **修复**：trick有分牌时出最大主牌，无分牌时出最小主牌
- **影响**：减少大王/小王浪费

### ✅ 问题3：末家"主分将吃"在队友确定最大时浪费主牌
- **修复**：删除末家队友确定最大时的主分将吃选项
- **影响**：保留主牌用于后续管住对手

### ✅ 问题4：没有利用CardTracker的缺门信息
- **修复**：首家出牌时通过`isPlayerVoidInSuit`检测对手缺门，降低缺门花色的出牌评分
- **影响**：避免出对手缺门的花色被将吃

### ✅ 问题6：没有考虑手牌数量对策略的影响
- **修复**：添加手牌数量因子，后期(+5)更激进，前期(-5)更保守
- **影响**：策略更灵活

### ✅ 问题7：副牌出牌顺序不合理
- **修复**：
  - A单张评分110（最大的牌，优先出）
  - K/10/5分牌需要A保护才安全出，否则评分低(35-45)
  - A已出过后，K/10/5变为最大牌，可以安全出
  - 非分牌(Q,J,9,8,7,6,4,3)从大到小出，安全无分牌损失
- **影响**：确保不会在没有保护的情况下送出分牌

### ✅ 问题10：扣底策略对主牌分牌处理粗糙
- **修复**：主牌分牌优先级-20（比非分主牌-50更容易被扣掉）
- **影响**：减少被抠底时的分数损失

### ✅ 问题11：跟牌策略中"最小分牌(无奈)"评分过高
- **修复**：评分从75/68降到65/60
- **影响**：确保有非分牌可选时不会误选分牌

### ✅ 问题13：AI没有保底意识
- **修复**：
  - `analyzeGameState` 新增 `bottomCardPoints`（底牌实际分值）和 `bottomProtection`（保护等级0/1/2）
  - 庄家方：底分≥30(后期)时大王评分20→75，小王评分50→70
  - 闲家方：估算底分≥25(后期)时小王评分50→35，小主牌评分50→35（保留大牌抢底）
  - 跟牌：闲家方有强保底需求时，始终出最小能赢的主牌
- **影响**：庄家方底分多时用大牌保底，闲家方底分多时保留大牌抢底

## 十一、未修复问题（低优先级）

### 问题5：甩牌策略可能不合法
- **原因**：需要复杂的合法性验证，`validatePlay`会拒绝不合法的甩牌
- **影响**：中等，被拒绝后会走fallback

### 问题8：将吃风险评估不足
- **原因**：需要复杂的概率计算，评估下家是否有更大的主牌
- **影响**：中等，当前策略已经根据分牌价值调整

### 问题9：`ensureCorrectPlayCount` 可能改变策略意图
- **原因**：结构性问题，需要重构策略函数生成正确数量的候选
- **影响**：低，补齐逻辑优先补同花色牌

### 问题12：没有"拆对手对子"的策略
- **原因**：需要推断对手手牌，非常复杂
- **影响**：低，属于高级战术
