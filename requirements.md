# 打升级（拖拉机/80分）联机游戏 — 需求报告（精简版）

---

## 一、游戏规则体系

### 1.1 基本架构

| 项目 | 说明 |
|------|------|
| 人数 | 4人（固定），2v2对家合作 |
| 牌数 | 2副/3副/4副牌模式 |
| 座位 | 东南西北四方，对家为队友（1-3一家，2-4一家） |
| 目标 | 抢分升级，先打过A的一方获胜 |

### 1.2 牌型与大小规则

**主副区分：**
- 每局通过亮主确定主牌花色（黑桃/红桃/方块/梅花），剩余三门为副牌
- 无主模式：所有花色均为副牌，仅大小王和级牌为主
- 级牌：当前庄家级别的牌（如打2时，所有2为主牌）

**主牌大小顺序（以2副牌为例）：**
```
大王 > 小王 > 主花色级牌 > 其他花色级牌 > A > K > Q > J > 10 > 9 > 8 > 7 > 6 > 5 > 4 > 3
```

**牌型分类：**

| 牌型 | 说明 | 示例 |
|------|------|------|
| 单张 | 任意一张牌 | 红桃A |
| 对子 | 两张同点数同花色 | 两张红桃A |
| 拖拉机 | 相邻点数的连对（主/副均可） | 红桃KK+红桃QQ（主牌拖拉机） |
| 甩牌 | 同时出多张符合规则的牌 | 需确认是否为同花色 |
| 四张/多张 | 3副及以上时的多张相同牌 | 三张红桃A |

**拖拉机规则：**
- 必须是同花色、相邻点数、且均为对子/多张
- 级牌参与时，级牌与其他主牌可以组成拖拉机（如主2+主A可连）
- 小王+大王可组成拖拉机
- 副牌拖拉机不能跨越级牌断点（如打2时，副牌A和K不能连，因为级牌2夹在中间）

### 1.3 亮主与反主规则（方案A）

| 牌副数 | 亮主条件 | 加固条件 | 备注 |
|--------|----------|----------|------|
| **2副牌** | 2张同色级牌 | 无加固 | 反主需2张其他花色级牌，或大小王对 |
| **3副牌** | 2张同色级牌 | 3张同色级牌 | 加固极少见 |
| **4副牌** | 2张同色级牌 | 3张或4张同色级牌 | 各地规则不一 |

**反主优先级：** 单张级牌 < 对级牌 < 王对 < 无主反（大小王对）

### 1.4 出牌与得分规则

**得分牌：**
- 5 = 5分，10 = 10分，K = 10分
- 每副牌共100分，多副牌总分 = 100 × 牌副数

**毙牌规则：**
- 首家出副牌，若某玩家无此花色，可用主牌毙
- 多家毙牌时，主牌大者获胜

**甩牌规则：**
- 首家可甩牌（一次性出多张同花色组合牌）
- 甩牌必须保证无人能大过其中任何一部分，否则判定为"违规甩牌"
- **违规甩牌处理（已确认）：** 系统自动识别违规甩牌，自动将甩牌拆成最小牌组合打出

### 1.5 升级规则

| 得分情况（2副牌总分200） | 结果 |
|----------|------|
| 0分（大光） | 庄家方升3级 |
| 5-35分 | 庄家方升2级 |
| 40-75分 | 庄家方升1级 |
| 80-115分 | 换庄，不升级（庄下） |
| 120-155分 | 闲家方升1级（反庄） |
| 160-195分 | 闲家方升2级 |
| 200分（小光） | 闲家方升3级 |

**多副牌升级阈值按比例调整。**

**过A规则（已确认）：** 必须打A且获胜（闲家得分未达庄下标准）才算过A成功；若庄下则留在A继续打，不退回。

### 1.6 扣底规则（已确认）

- 庄家扣底牌后，若闲家最后一轮用拖拉机抠底，底分翻倍
- **翻倍倍数 = 抠底拖拉机所包含的牌张数**
- 例如：用3344（4张牌）抠底，底分翻4倍；用334455（6张牌）抠底，底分翻6倍

### 1.7 荒庄处理（已确认）

- 发牌至倒数第二张时暂停，系统询问所有玩家是否亮主
- 若有人亮主，正常进入反主阶段
- 若无人亮主，则发最后一张牌，以最后一张牌的花色作为本局主牌
- 无主情况：最后一张牌为级牌或其他特殊处理时，按具体牌面规则判定

## 二、功能需求

### 2.1 用户系统（简化版）

| 模块 | 功能点 | 优先级 |
|------|--------|--------|
| 账号体系 | 微信扫码登录（主要）+ 游客登录 | P0 |
| 用户资料 | 昵称、头像 | P0 |
| 好友系统 | 添加好友、邀请对战 | P1 |

**登录流程：**
- 微信扫码登录：浏览器展示二维码，用户微信扫码授权后直接进入游戏
- 游客登录：无需注册，自动生成临时账号，可后续绑定微信
- 首次登录自动创建用户，无需额外注册步骤

### 2.2 游戏房间系统

| 模块 | 功能点 |
|------|--------|
| 快速匹配 | 根据段位匹配4人局 |
| 创建房间 | 房主自定义规则 |
| 房间号加入 | 输入6位房间号加入 |
| 托管功能 | 玩家离线/超时自动出牌 |

**房间配置项：**
- 牌副数：2副/3副/4副
- 过A规则：必须大光/只须保庄/允许背靠背
- 甩牌校验：严格模式（自动拆最小牌）/ 宽松模式（允许人工确认）
- 出牌超时：15秒/20秒/30秒
- 聊天开关：文字/语音/全部关闭

### 2.3 聊天系统

| 类型 | 功能 | 默认状态 |
|------|------|----------|
| 文字聊天 | 快捷短语 + 自定义输入 | 开启 |
| 实时语音 | WebRTC实时语音（浏览器原生） | **默认开启，可手动关闭** |
| 语音消息 | 短语音消息收发 | 开启 |
| 表情包 | 游戏内表情互动 | 开启 |

**实时语音方案（已确认）：**
- 使用浏览器原生 **WebRTC** 实现P2P或SFU架构实时语音
- 默认进入房间自动开启麦克风，玩家可手动静音
- 无需额外SDK，纯Web技术实现
- 备选：声网Agora Web SDK（稳定性更高，按量计费）

### 2.4 Web端适配需求

| 维度 | 需求 |
|------|------|
| 平台 | **纯HTML5 Web端**，浏览器直接访问 |
| PWA支持 | 可添加到手机主屏幕，接近原生App体验 |
| 横竖屏 | 支持横屏（推荐）+ 竖屏自适应 |
| 操作方式 | 点击选牌 + 滑动多选 + 双击出牌 |
| 浏览器兼容 | Chrome/Safari/Edge/微信内置浏览器 |
| 性能要求 | 中低端手机流畅运行 |
| 流量消耗 | 1小时游戏 < 30MB（不含语音） |

## 三、技术架构建议

### 3.1 推荐技术栈（Web端）

| 层级 | 推荐方案 |
|------|----------|
| 前端框架 | HTML5 + CSS3 + Vanilla JavaScript / Vue 3 / React |
| 渲染方式 | Canvas 2D 或 DOM + CSS Transform（牌面动画） |
| 通信协议 | WebSocket（浏览器原生） |
| 数据格式 | JSON 或 Protocol Buffers |
| 后端语言 | Go（高并发）或 Node.js |
| 数据库 | PostgreSQL / MySQL + Redis |
| 实时语音 | WebRTC（P2P/SFU） |
| 登录认证 | 微信OAuth 2.0 |
| 部署 | Nginx + Docker |

### 3.2 前端技术选型分析

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **Vanilla JS + Canvas** | 轻量、无依赖、性能最好 | 开发效率较低 | ★★★★☆ |
| **Vue 3 + DOM** | 开发效率高、组件化、生态好 | 大量DOM节点可能影响性能 | ★★★★★ |
| **Phaser.js** | 游戏专用引擎、动画丰富 | 学习成本、体积较大 | ★★★☆☆ |

**推荐：** Vue 3 + Canvas 混合方案——UI层用Vue组件，牌桌渲染用Canvas。

### 3.3 游戏状态同步

采用**状态同步**为主，服务端作为唯一真理源。出牌动作可客户端预表现，服务端校验后广播结果。

### 3.4 微信扫码登录流程

```
用户访问游戏页 → 展示微信登录二维码
    ↓
用户微信扫码 → 微信回调授权Code
    ↓
后端用Code换取OpenID/UnionID → 查询或创建用户
    ↓
返回JWT Token → 前端存储 → 进入游戏大厅
```
